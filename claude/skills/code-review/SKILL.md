# コードレビュー

4つの専門レビュアーを並列実行し、多角的にコードを検査するワークフロー。

## レビュー対象の特定

ユーザーが明示的に対象を指定しない場合、以下の順で特定する:

1. PR番号が指定された場合 → そのPRの変更ファイル
2. ステージングされた変更がある場合 → `git diff --cached`
3. 未コミットの変更がある場合 → `git diff`
4. 直近のコミット → `git diff HEAD~1`

## 並列レビューの実行

以下の4つのサブエージェントを**同時に**起動する:

1. **design-reviewer** — コード品質、設計原則、命名、責務分離、既存パターンとの一貫性
2. **security-reviewer** — 脆弱性、OWASP Top 10、インジェクション、認証・認可、データ露出
3. **test-reviewer** — テストカバレッジ、不足テスト、アサーション品質、エッジケース
4. **performance-reviewer** — N+1クエリ、メモリリーク、不要な計算、スケーラビリティ

各サブエージェントには、レビュー対象のファイルパスと変更内容を渡す。

## 結果の統合

全レビュアーの結果を以下の形式に統合する:

### 報告フォーマット

```
## レビュー結果サマリー

### Critical / Must Fix
（セキュリティ脆弱性、バグ、重大な設計問題）

### Should Fix
（設計改善、パフォーマンス問題、テストギャップ）

### Nits
（命名、スタイル、軽微な改善）

### 良い点
（設計が適切な箇所、品質の高い部分）
```

- 重複する指摘はマージする
- 複数レビュアーが同じ箇所を指摘した場合、最も重要度の高い指摘に統合
- 指摘がない場合は「問題なし」と明記
